## 一、当前问题

现有 MEMORY.md 是个空壳，缺乏结构：
- 所有东西往一个文件塞，没有分层
- 没有反思机制，只记事实不回顾判断
- 没有检索优先级，启动时一股脑全读
- 私有记忆和共享记忆的边界模糊
- 没有生命周期管理，记忆只增不减
- 没有索引机制，Agent 无法按需检索
- Ally（知心伙伴）长期运行但未纳入团队架构

## 三、团队架构：4+1+1

原 4+1 架构升级为 **4+1+1**，增加独立的内在系统：

```
工作系统（创造）              生活系统（运维）        内在系统（成长）
┌────────────────────┐      ┌──────────┐          ┌──────────┐
│ Radar → Partner    │      │          │          │          │
│   ↓       ↓        │      │  Butler  │          │   Ally   │
│ Workshop → Keeper  │      │          │          │          │
└────────────────────┘      └──────────┘          └──────────┘
         ↑                        ↑                    ↑
      你(创作者)             你(被照顾的人)        你(成长中的人)
```

三个系统各自独立，互不交互：
- **工作系统**：你是主角，agent 是团队
- **生活系统**：你是甩手掌柜，Butler 打理一切
- **内在系统**：你是对话者，Ally 是认知镜子

### Ally 独立的理由（第一性原理检验）

| 检验维度 | Ally vs Partner | Ally vs Butler |
|---------|----------------|----------------|
| 上下文冲突 | 是。Partner 需要工作决策上下文，Ally 需要情感/认知/生活上下文 | 是。Butler 管事务，Ally 管内在 |
| 自主权不同 | 是。Partner 输出 brief，Ally 输出洞察 | 是。Butler 高自主（自动安排），Ally 跟着你的节奏走 |
| 时间节奏不同 | 是。Ally 可能在任何时刻被需要 | 是。Butler 定时推送，Ally 随时响应 |
| 失败代价不同 | 是。Partner 错了影响工作方向，Ally 错了影响情绪和认知——更私密、更不可逆 | 是。Butler 错了重新安排，Ally 错了可能强化盲点 |

### Ally 的定位

| 维度 | 定义 |
|------|------|
| 本质 | 认知镜子、成长伙伴 |
| 来源 | 基于"知心伙伴 v6.0"创建的 OpenClaw 实例，已长期运行验证 |
| 自主权 | 低——必须跟着亚臣的节奏，不主动发起 |
| 时间节奏 | 随时响应，无固定 heartbeat |
| 输出物 | 洞察、情感解读、认知引导、盲点揭示 |
| 核心价值 | 看见、接纳、不评判；把模糊感受变清晰语言，砸碎认知枷锁 |
| 渗透范围 | 跨工作和生活——内省本身不区分场景 |

## 四、私有记忆架构（每个 Agent）

```
Agent目录/
├── .abstract              ← L0 索引（100-300 tokens，启动时第一个读）
├── MEMORY.md              ← L1 长期稳定认知（带 P0/P1/P2 生命周期标签）
├── SESSION-STATE.md       ← 当前工作缓冲区（heartbeat 覆写，context compaction 救生筏）
├── reflections/           ← L2 原始反思日志
│   ├── 2026-03-10.md
│   └── archive/           ← 已提炼的过期反思
└── lessons/               ← 结构化经验教训（从 reflections 提炼）
```

### 各文件职责

| 文件 | 性质 | 谁写 | 谁读 | 更新频率 |
|------|------|------|------|---------|
| `.abstract` | 记忆目录页（索引） | 自己 | 自己 | 每周/每月刷新 |
| `MEMORY.md` | 长期稳定认知 | 自己 | 只有自己 | 每周从 reflections/lessons 提炼 |
| `SESSION-STATE.md` | 当前工作状态 | 自己 | 只有自己 | 实时覆写 |
| `reflections/` | 原始反思日志 | 自己 | 只有自己 | session 驱动 |
| `lessons/` | 结构化经验 | 自己 | 只有自己 | 从 reflections 提炼 |

### 三层私有记忆的硬边界

三层之间**只晋升结论，不复制内容**。晋升后，下层原文可归档或删除。

| 层 | 存什么 | 不存什么 | 检验问题 |
|---|--------|---------|---------|
| `reflections/` | 原始观察：当天发生了什么、我注意到什么、我的即时反应 | 不存结论、不存规则 | "这是今天才观察到的吗？" → 是则写这里 |
| `lessons/` | 可复用经验：带上下文的操作指南，说清"在什么情况下、做什么、为什么" | 不存原始事件细节，不存脱离了上下文也成立的普适认知 | "下次遇到类似情况，翻这篇能直接用吗？" → 是则写这里 |
| `MEMORY.md` | 长期稳定认知：脱离了具体事件仍然成立的原则、偏好、模式 | 不存操作步骤，不存具体案例 | "即使忘了所有具体事件，这条认知还成立吗？" → 是则写这里 |

**举例说明流转过程：**

```
reflections/2026-03-10.md:
  "今天给哈恩写详情页文案，第一版被亚臣打回，原因是首屏没有直接点出核心功效，
   而是用了一段品牌故事铺垫。亚臣说'用户 3 秒内看不到功效就划走了'。"

  ↓ 提炼为 lesson（保留上下文，可复用）

lessons/详情页首屏经验.md:
  "电商详情页首屏必须在前 3 秒内传达核心功效，品牌故事铺垫放在第二屏之后。
   原因：用户注意力窗口极短，首屏是唯一的留人机会。来源：哈恩详情页项目。"

  ↓ 提炼为 MEMORY.md 条目（脱离上下文的稳定认知）

MEMORY.md #P0:
  "亚臣的交付偏好：所有面向消费者的内容，功效信息必须前置，品牌叙事后置。"
```

**反重复规则：** 一条认知晋升到上层后，下层不再保留同义表述。如果 lessons/ 中的某条经验已被完整提炼进 MEMORY.md，该条目从 lessons/ 中删除或标记为"已晋升"。

## 五、生命周期管理（P0/P1/P2）

MEMORY.md 中每条记忆带生命周期标签和验证时间戳：

| 级别 | 含义 | 有效期 | 过期处理 |
|------|------|--------|---------|
| **P0** | 永久（身份、核心偏好、长期原则） | 永不过期 | 仅人工修正 |
| **P1** | 活跃项目/阶段性认知 | 90 天 | 到期降级或归档 |
| **P2** | 临时/待验证 | 30 天 | 到期归档 |

### MEMORY.md 条目格式

```markdown
# MEMORY.md — Radar

> 最后提炼：2026-03-10 | 版本：v1

## P0 — 永久
- 亚臣关注领域权重：品牌设计 > 营销策略 > AI 工具 > 行业趋势
  <!-- verified: 2026-03-08 -->

## P1 — 90天
- 哈恩品牌当前处于详情页优化阶段，重点关注竞品视觉差异
  <!-- verified: 2026-03-05 | expires: 2026-06-05 -->

## P2 — 30天
- 某竞品最近在调整主图风格，可能是测试阶段
  <!-- verified: 2026-03-09 | expires: 2026-04-09 -->
```

超过有效期未被验证/引用的条目 → 降级或归档到 `reflections/archive/`。

| Agent | P0 长期认知 | P1/P2 阶段性 | 不存什么 |

| **Radar** | 信息源质量评估、亚臣的关注领域权重、过滤经验 | 当前监控重点、近期发现的信息源 | 具体新闻内容（→ 共享层竞品档案） |

**Butler：**

| 永不落盘 | 可落盘 |
|---------|--------|
| 银行卡号、身份证号、具体金额、密码、API Key | 支出分类比例、资产配置结构、消费趋势、日期 |

**Ally：**

| 永不落盘 | 可落盘 |
|---------|--------|
| 涉及第三方的私密细节、他人隐私 | 情绪模式、认知转变、盲点类型、成长节点（脱敏后） |

## 七、.abstract 索引机制

每个 Agent 的 `.abstract` 文件是记忆系统的"目录页"，控制在 100-300 tokens 以内。

### 格式示例

```markdown
# .abstract — Radar

## 记忆概览
- MEMORY.md: 3条P0(信息源/关注领域/过滤经验) + 2条P1(活跃项目) + 1条P2
- SESSION-STATE.md: 当前监控任务状态
- reflections/: 最近7天有5篇，主题集中在竞品动态
- lessons/: 2篇（信息源筛选经验、误报处理）

## 关键词索引
信息源质量 → MEMORY.md#P0
竞品动态 → reflections/2026-03-09.md, lessons/竞品监控经验.md
```

**核心原则：Agent 启动时先读 .abstract，再按需决定读什么，不是一股脑全读。**

### 30_交付物存档/ 准入规则

交付物存档不是垃圾场。只有以下两类内容可以进入：

1. **最终交付版** — 经 Keeper 审核通过、亚臣确认的终稿
2. **典型案例** — 被明确判定为"值得复用的参考案例"的交付物，需标注复用价值

**不进入的内容：** 中间稿、探索性方案、被否决的版本、未经审核的草稿。这些留在 Workshop 本地或直接丢弃。

每个存档文件的 frontmatter 必须包含：

```yaml
---
type: 最终交付版 / 典型案例
project: [项目名]
date: YYYY-MM-DD
reuse_value: [一句话说明为什么值得存档]
---
```

### 品牌规范的双存在

- **Obsidian 文档 = 知识态**（人可读、可演化、版本化的源头）
- **Skill = 执行态**（给 AI 用的 prompt 规则，从文档编译而来）
- **流向：Obsidian 文档 → 成型后编译为 skill**

## 九、记忆检索协议

## 十、反思机制

### 触发方式

| Agent | 触发方式 | 理由 |
|-------|---------|------|
| **Radar** | heartbeat 末尾挂载 | 已有定时触发 |
| **Butler** | heartbeat 末尾挂载 | 已有定时触发 |
| **Partner** | 每次有实质交互的 session 结束 | 按需触发，无事不记 |
| **Workshop** | 每次有实质交互的 session 结束 | 按需触发，无事不记 |
| **Keeper** | 每次有实质交互的 session 结束 | 按需触发，无事不记 |
| **Ally** | 每次有实质交互的 session 结束 | 按需触发，无事不记 |

**最小信息量门槛：** 模板中至少有一个区块有实质内容才写。无实质内容时不写。

### 反思模板（工作系统 + 生活系统）

```markdown
# YYYY-MM-DD 反思

## 信号强度：P0 / P1 / P2

## 意外（与预期不符的事）
- ...

## 修正（规则/标准需要调整）
- ...

## 沉淀（值得写入 MEMORY.md 的结论）
- ...  <!-- 写入前先读 MEMORY.md 检查重复/冲突 -->
```

### Ally 专用反思模板

```markdown
# YYYY-MM-DD 反思

## 信号强度：P0 / P1 / P2

## 认知变化（今天亚臣的认知有什么移动）
- ...

## 模式观察（反复出现的思维/情绪模式）
- ...

## 盲点线索（可能存在但尚未揭示的盲点）
- ...

## 沉淀（值得写入 MEMORY.md 的结论）
- ...  <!-- 写入前先读 MEMORY.md 检查重复/冲突 -->
```

### 流转机制

```
session 驱动 → reflections/YYYY-MM-DD.md（原始记录）
每周        → 回顾本周 reflections，提炼到 lessons/ + MEMORY.md
每月        → 扫描 P1/P2 过期条目 → archive/；刷新 .abstract
```

### CRUD 验证规则

写入 MEMORY.md 前必须：
1. 先读取现有内容
2. 检查是否重复（已有相同条目）
3. 检查是否冲突（与已有条目矛盾）
4. 冲突时：暴露冲突，不静默覆盖

## 十二、知识管线协议（Engine 接入）

### 背景

亚臣的 Obsidian 知识引擎（`01_Engine/`）是独立于工作交付管线的个人知识炼金管道：

```
Cubox (采集) → Learning (消化) → Method (提纯) → Writing (输出)
```

Engine 不归属于任何 agent。它是亚臣的个人领地，agent 团队是它的协作者和调用者。现有 agent 不需要重建，而是在工作管线之外增加一条知识管线：

```
工作管线：  Radar信号 → Partner推演 → Workshop交付 → Keeper审核 → 客户
               ↕            ↕            ↕            ↕
知识管线：  Cubox采集 → Learning消化 → Method提纯 → Writing输出 → 个人影响力
```

两条管线共享同一组 agent，切换靠上下文区分（对话开头明确是工作任务还是知识任务），不靠切换 agent。

### 运转模式 × 加工阶段矩阵

|  | Cubox 采集 | Learning 消化 | Method 提纯 | Writing 输出 |
|--|-----------|--------------|------------|-------------|
| **Radar（感知）** | ★ 投喂素材 | | | |
| **Partner（判断）** | | ★ 协作消化 | ★ 辅助提纯 | |
| **Workshop（执行）** | | | | ★ 生产内容 |
| **Keeper（校验）** | | | ★ 审核可复用性 | ★ 审核发布质量 |
| **Ally（内省）** | | ★ 认知层消化 | ★ 经验提炼 | |
| **Butler** | 不参与 | 不参与 | 不参与 | 不参与 |

### 各层详细规则

#### Cubox（采集层）

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **你** | 主体 | 随手采集文章、灵感、截图 |
| **Radar** | 投喂者 | 扫描中发现的高价值长期素材，标记后流入 Cubox |

**Radar 投喂规则：**
- Radar 的日常情报简报仍然输出到工作管线（共享层 `10_竞品档案/`）
- 当 Radar 判断某条信息的价值超越了即时竞品情报、值得长期沉淀时，标记为"建议入库 Engine"
- 最终是否入库由亚臣确认（Radar 不直接写入 Cubox）

#### Learning（消化层）

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **你** | 主体 | 阅读、研习、理解 |
| **Partner** | 讨论伙伴 | 陪你拆解方法论、讨论书中观点、推演概念的适用边界 |
| **Ally** | 反思伙伴 | 帮你从经历中提炼认知、看见学习过程中的盲点和偏见 |

**与工作管线的区别：**
- Partner 在工作管线输出 brief；在知识管线输出"我们对这个概念的共同理解"
- Ally 在知识管线的参与聚焦"你在学什么"而不是"你学到了什么"——关注学习过程中暴露的认知模式

#### Method（提纯层）

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **你** | 主体 | 从 Learning 中提炼框架/SOP/Prompt，脱水晋升 |
| **Partner** | 质量讨论 | "这个框架的适用边界在哪？有没有遗漏的场景？" |
| **Keeper** | 可复用性审核 | "这个 SOP 够不够清晰？换一个人能直接用吗？" |
| **Ally** | 经验提炼 | 帮你从反复出现的经验中提炼出稳定认知 |

**核心原则：** Method 层的主体必须是你。Agent 辅助质检和讨论，但"什么值得提纯为方法论"的判断权在你。

#### Writing（输出层）

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **你** | 主体 | 确定选题、定调性、最终审稿 |
| **Workshop** | 生产者 | 执行内容生产（公众号文章、社媒内容、行业观察） |
| **Keeper** | 审核者 | 发布前审核内容质量、事实准确性、品牌一致性 |

**与工作管线的区别：**
- 工作管线的 Workshop 输出客户交付物（详情页、包装文案），受品牌规范和客户 brief 约束
- 知识管线的 Workshop 输出个人内容（文章、观点），受亚臣的个人表达风格和价值观约束
- Workshop 需要在 session 开头识别"这是客户任务还是个人创作"，加载不同的上下文和标准

### Engine 的读写权限

| Agent | 10_Cubox | 20_Learning | 30_Method | 40_Writing |
|-------|----------|-------------|-----------|------------|
| **Radar** | 建议入库（需亚臣确认） | 不参与 | 只读（取弹药） | 不参与 |
| **Partner** | 不参与 | 读写（协作消化） | 读写（辅助提纯） | 不参与 |
| **Workshop** | 不参与 | 只读（取素材） | 只读（取弹药） | 读写（生产内容） |
| **Keeper** | 不参与 | 不参与 | 只读（审核可复用性） | 只读（审核发布质量） |
| **Ally** | 不参与 | 读写（认知层消化） | 读写（经验提炼） | 不参与 |
| **Butler** | 不参与 | 不参与 | 不参与 | 不参与 |

**注意：** 所有 agent 对 Engine 的写入都在亚臣的 session 中发生，不存在 agent 独立写入 Engine 的场景（Radar 的入库建议也需确认）。Engine 是你的个人领地，agent 是协作者不是管理者。

### 知识管线的上下文切换

Agent 通过对话开头的上下文判断当前在哪条管线：

| 信号 | 判定为 |
|------|--------|
| "帮我看看这个竞品""给哈恩写个文案" | 工作管线 |
| "我在读一本书""帮我整理下这个方法论""写篇公众号文章" | 知识管线 |
| 不确定 | 先响应，再轻量确认"这是要落到项目里，还是你自己的学习研究？" |

## 十三、创意管线协议（Lab 接入）

### 背景

亚臣的创意实验室（`04_Lab/`）是独立于工作管线和知识管线的第三条管线。三条管线的本质区别：

```
工作管线：  信号 → 推演 → 交付 → 审核        节奏：任务驱动，有 deadline
知识管线：  采集 → 消化 → 提纯 → 输出        节奏：持续积累，线性递进
创意管线：  捕捉 → 碰撞 → 孵化 → 沉淀        节奏：随机涌现，允许沉睡
```

Engine 是知识炼金——线性的、确定性递增的，每一步都比上一步更精炼。Lab 是创意孵化——非线性的、允许混乱和失败的，种子可能沉睡三个月后被另一个种子唤醒。

Lab 同样不归属于任何 agent，是亚臣的个人创意领地。

### Lab 的四层漏斗

```
10_灵感池 (Capture) → 20_实验项目 (Incubation) → 30_视觉中台 (Production) → 99_资源与工具 (Infrastructure)
      ↓                      ↓                         ↓                          ↓
   原子捕捉              原型孵化                  成熟资产                    底层基建
   允许混乱              允许失败                  即插即用                    通用组件
```

### 运转模式 × 创意阶段矩阵

|  | 灵感池（捕捉） | 灵感池（碰撞） | 实验项目（孵化） | 视觉中台（沉淀） |
|--|--------------|--------------|----------------|----------------|
| **Radar** | | | | |
| **Partner** | | ★ 逻辑碰撞 | ★ 方向评估 | |
| **Workshop** | | | ★ 原型制作 | ★ 资产打磨 |
| **Keeper** | | | | ★ 质量审核 |
| **Ally** | ★ 灵感捕捉 | ★ 感受碰撞 | | |
| **Butler** | 不参与 | 不参与 | 不参与 | 不参与 |

### 各层详细规则

#### 灵感池 — 捕捉

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **你** | 主体 | 随时随地捕捉灵感，使用 `/capture` 生成种子文档 |
| **Ally** | 灵感触发者 | 在内省对话中识别创意闪念、诗意意象、身体感受，主动建议捕捉为种子 |

**为什么是 Ally 而不是 Radar：** 灵感是从内在涌现的，不是从外部扫描到的。Ally 在对话中最容易接触到亚臣的创意闪念——一个隐喻、一个意象、一段身体感受——这些恰好是种子的原料。

**Ally 的捕捉原则：**
- 对话中发现有诗意或创意潜力的表述时，主动建议"这个值得捕捉为种子"
- 不替亚臣判断种子的价值——捕捉阶段允许混乱，筛选是后面的事
- 捕捉后自动提取 4D 标签（领域/媒介/形式/情绪）

#### 灵感池 — 碰撞

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **Partner** | 逻辑碰撞 | "这两个种子之间有没有结构性联系？能不能组合成一个更大的概念？" |
| **Ally** | 感受碰撞 | "这两个意象放在一起让你想到什么？它们的情感张力在哪？" |

**两种碰撞的区别：** Partner 看结构（这两个东西能不能拼成一个项目），Ally 看感受（这两个东西放在一起有没有化学反应）。这正好防止 Ally 漂向 Partner——在碰撞实验室里两者的视角天然不同。

#### 实验项目 — 孵化

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **你** | 主体 | 决定哪个种子值得孵化、项目方向怎么定 |
| **Partner** | 方向评估 | 帮你评估孵化方向的可行性和价值，但不做最终判断 |
| **Workshop** | 原型制作 | 执行原型（视觉稿、文字稿、视频脚本、角色设定） |

**与工作管线的区别：** Workshop 在 Lab 里的质量标准更宽松。工作管线追求"可交付"，Lab 追求"有意思"。允许粗糙、允许实验性、允许失败。

#### 视觉中台 — 沉淀

| 参与者 | 角色 | 具体动作 |
|--------|------|---------|
| **Workshop** | 资产打磨 | 把验证过的实验成果打磨成可复用的成熟资产（Prompt、工作流、视觉模板） |
| **Keeper** | 质量审核 | 审核资产的完成度和可复用性——但标准是"创意完成度"而非"合规性" |

**这是 Lab 和 Work 的桥梁：** 视觉中台的成熟资产可以直接被工作管线调用，支撑 `03_Work` 中的商业项目。

### Lab 的读写权限

| Agent | 10_灵感池 | 20_实验项目 | 30_视觉中台 | 99_资源与工具 |
|-------|----------|-----------|-----------|-------------|
| **Radar** | 不参与 | 不参与 | 不参与 | 不参与 |
| **Partner** | 只读（碰撞建议） | 读写（方向评估） | 不参与 | 不参与 |
| **Workshop** | 不参与 | 读写（原型制作） | 读写（资产打磨） | 只读（取工具） |
| **Keeper** | 不参与 | 不参与 | 只读（质量审核） | 不参与 |
| **Ally** | 读写（灵感捕捉+碰撞） | 不参与 | 不参与 | 不参与 |
| **Butler** | 不参与 | 不参与 | 不参与 | 不参与 |

**注意：** 与 Engine 相同，所有 agent 对 Lab 的写入都在亚臣的 session 中发生。Lab 是你的创意领地，agent 是协作者。

### 三条管线之间的流动

```
         Engine                    Lab                     Work
      ┌──────────┐           ┌──────────┐           ┌──────────┐
      │  Method   │──弹药──→ │ 实验项目  │           │          │
      │  提纯层   │←─方法论─ │  孵化区   │           │ Workshop │
      └──────────┘           └────┬─────┘           └────┬─────┘
                                  │                      │
                             视觉中台                     │
                             成熟资产 ──── 即插即用 ──────→┘
```

| 流动方向 | 具体场景 |
|---------|---------|
| Engine → Lab | Method 里的设计框架用于指导 Lab 的创意实验 |
| Lab → Engine | 实验中验证出的新工作流/Prompt 沉淀为 Method 里的 SOP |
| Lab → Work | 视觉中台的成熟资产直接支撑工作管线的商业交付 |
| Work → Lab | 工作中发现的创意火花回流到灵感池等待孵化 |

### 创意管线的上下文切换

| 信号 | 判定为 |
|------|--------|
| "我有个灵感""捕捉一下这个""这两个种子碰一碰" | 创意管线 — 灵感池 |
| "棱镜计划推进一下""数字人类的角色设定""做个原型试试" | 创意管线 — 实验项目 |
| "这个 Prompt 可以沉淀了""打磨成可复用的" | 创意管线 — 视觉中台 |
| 不确定是创意实验还是工作任务 | 先响应，再确认"这是实验性质的，还是要落到项目交付里？" |
