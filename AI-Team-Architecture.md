# AI 团队架构总纲

> AI 时代的分工不再是"你是谁"（角色），而是"你能碰什么"（上下文）和"你能决定什么"（权限）。

---

## 一、第一性原理：为什么要分？

只有四个理由值得把一个 agent 拆成两个。如果两件事在以下四个维度上都相同，就不该拆开——拆开只会增加协调成本。

| 检验维度 | 问题 |
|---------|------|
| **上下文冲突** | 两件事需要的背景信息是否互相干扰？ |
| **自主权不同** | 一件事可以放手让它做，另一件是否不行？ |
| **时间节奏不同** | 一件事是 24/7 持续的，另一件是否按需触发？ |
| **失败代价不同** | 一件做错了可以重来，另一件是否不可逆？ |

---

## 二、分工维度已变

| 旧维度（按角色） | 新维度 |
|---|---|
| 品牌设计师 / 包装设计师 | **上下文边界** — 这个 agent 需要接入什么信息？ |
| 文案 / 设计 / 运营 | **决策权限** — 它能自主做什么决定？ |
| 初级 / 高级 / 总监 | **时间尺度** — 它处理即时任务还是长期战略？ |
| 甲方 / 乙方 | **风险容忍度** — 做错了代价多大？ |

**关键洞察：** 能力可以随时加载（Skills/Tools），但人格必须恒定（SOUL.md）。分的不是岗位，是运转模式。

---

## 三、角色 vs 灵魂

传统时代，角色提供一致性——"我是品牌设计师"帮你决定关注什么、忽略什么。

AI 打碎角色后（一个 bot 既能设计又能写文案），一致性从哪来？

**答案是 SOUL.md。** 它不定义"你是设计师"，而定义：
- 你做事的调性（简洁还是啰嗦？大胆还是谨慎？）
- 你对外部操作的态度（先请示还是先做了再说？）
- 你和用户的关系（工具？助手？合作者？）

> 角色是工业时代的一致性方案，灵魂是 AI 时代的一致性方案。

## 四、4 + 1 架构

### 工作系统（创造）

四种不可混淆的时态：

```
Radar → Partner → Workshop → Keeper
  │        │         │          │
感知信号  共创判断   生产交付   风险校验
  │        │         │          │
  └──→ 你 ────→ 你可中途介入 ──→ 你最终拍板
```

| Agent | 本质 | 自主权 | 时间节奏 | 输出物 |
|-------|------|--------|---------|--------|
| **Radar**（雷达） | 信息触角 | 高 | 24/7 持续 | 每日简报 + 异常警报 |
| **Partner**（合伙人） | 思维对手 | 低 | 按需对话 | Brief / 方向 / 框架 |
| **Workshop**（工坊） | 生产引擎 | Brief 内高、Brief 外零 | 任务驱动 | 成品交付物 |
| **Keeper**（门卫） | 质量守门人 | 审核自主、标准需确认 | 被触发式 | 审核报告 |

### 生活系统（运维）

| Agent | 本质 | 自主权 | 时间节奏 | 输出物 |
|-------|------|--------|---------|--------|
| **Butler**（管家） | 生活管家 | 高（金钱操作除外） | 24/7 + 定时推送 | 晨间简报 + 周回顾 |

### 两个系统的关系

```
工作系统（创造）          生活系统（运维）
┌──────────────────┐    ┌──────────┐
│ Radar → Partner  │    │          │
│   ↓       ↓      │    │  Butler  │
│ Workshop → Keeper│    │          │
└──────────────────┘    └──────────┘
        ↑                     ↑
        你                    你
   （创作者）            （被照顾的人）
```

左边你是主角，右边你是甩手掌柜。两个系统之间**不交互**。

## 五、系统规则

### 规则 1：Partner 不直接替代 Workshop
Partner 负责把事想明白，不负责长期埋头生产。Partner 的输出物永远是 brief，不是成品。

### 规则 2：Keeper 必须真有否决权
否则它只是礼仪，不是门卫。Keeper 的 SOUL.md 必须和 Workshop 存在价值观张力——Workshop 追求"做到最好"，Keeper 假设"它有问题"。

### 规则 3：Workshop 在关键节点强制暂停
不是"可以"介入，而是"必须"暂停等你确认。至少两个暂停点：
1. 发散 → 收敛（方向一旦定了很难回头）
2. 初稿完成 → 精修交付前（最后改内容的机会）

### 规则 4：Partner 当前不做调度
Partner 不指挥其他 agent。你才是中枢。当标准化流程积累足够多之后，可以逐步授权 Partner 触发已验证的流程。

### 规则 5：否决后有明确打回路径
- 执行层问题（文案措辞、数据错误）→ 打回 Workshop 重做
- 方向层问题（brief 本身有矛盾）→ 打回你 + Partner 重新讨论

## 六、共享记忆层

四个工作 agent 之间需要一份共享知识库（Butler 不参与）：
- 品牌规范（视觉 + 语言）
- 产品信息（配方、功效、法规）
- 竞品档案（Radar 持续更新）
- 历史决策记录（Partner 的重要对话结论）
- 审核标准（Keeper 的规则库）

**载体：** Obsidian Vault 或等效的结构化文件系统。所有 agent 可读，特定 agent 可写。

---

## 七、落地优先级

| 顺序 | Agent | 理由 |
|------|-------|------|
| 1 | **Radar** | 当前完全缺失，投入产出比最高 |
| 2 | **Partner** | 已有雏形（当前 Claude Code 用法），需要写更好的 SOUL.md |
| 3 | **Workshop** | 当"出活"成为瓶颈时再拆出来 |
| 4 | **Keeper** | 当交付质量成为痛点时再加 |
| 5 | **Butler** | 独立系统，随时可以开始，不阻塞工作系统 |

---

## 八、演化路径

```
现在 →    你 是中枢，手动调度一切
          Partner 是对话者

6个月 →   Partner 可触发标准化流程
          Workshop 可并行跑多个任务
          Keeper 建立了稳定的审核规则库

最终态 →  你只管两件事：
          定方向（什么值得做）
          判品质（做得够不够好）
          中间所有调度、执行、协调都自动化
```
